DEPLOYMENT GUIDE

Deploy AquaLife to Render

A step-by-step walkthrough for hosting this swimming pool membership app on Render's free plan.

i What you're deploying

Understand the app before you ship it.

AquaLife is a single unified Node application:

  • A Vite + React frontend that builds into a static dist/ folder.
  • An Express backend (server/index.js) that exposes the /api/* routes and serves the built frontend from the same process.
  • A SQLite database (better-sqlite3) that auto-creates and auto-seeds demo data on first boot.

Because the server serves both the API and the static site, you only need one Render Web Service — no separate frontend hosting required.

⚠ Important: the free plan has an ephemeral disk Render's free tier does not keep written files between restarts or deploys. The SQLite file (server/data.sqlite) will be wiped whenever the service restarts. The app automatically re-seeds fresh demo data when it finds an empty database, so this is fine for a proof-of-concept demo — but any members/check-ins created at runtime will not survive a restart. See Step 6 for how to add a persistent disk if you need that.

1 Push your code to a Git repository

Render deploys from GitHub, GitLab, or Bitbucket.

This project is not yet a Git repo. From the project folder (swim-mgr), run:

git init
git add .
git commit -m "Initial commit — AquaLife"

Then create an empty repository on GitHub and push to it:

git remote add origin https://github.com/<your-username>/swim-mgr.git
git branch -M main
git push -u origin main
Check your .gitignore The included .gitignore already excludes node_modules, dist, and the local server/data.sqlite* files. That is correct — Render installs dependencies and builds fresh on its own servers.

2 Create a new Web Service on Render

render.com → Dashboard → New.

  1. Log in at dashboard.render.com.
  2. Click New + → Web Service.
  3. Connect your GitHub account and select the swim-mgr repository.
  4. Render will open the service configuration screen — fill it in as shown in Step 3.
Pick a Web Service — not a Static Site Although the frontend is static, this app needs a running Node process for the Express API and the SQLite database. A "Static Site" cannot run server code.

Why these commands?

  • npm install — installs dependencies, and compiles the native better-sqlite3 module against Render's Linux environment.
  • npm run build — runs vite build to produce the dist/ folder the server will serve.
  • npm start — runs node server/index.js, which serves both the API and dist/ on a single port.
✓ The app already reads Render's PORT server/index.js uses process.env.PORT || 4000. Render injects PORT automatically, so the app binds correctly with no extra work.

4 Set the Node version (recommended)

Avoids native-build surprises with better-sqlite3.

Add an environment variable on the Environment tab so Render uses a known-good Node version:

KeyValue
NODE_VERSION20

Alternatively, commit a .node-version file or add an "engines" field to package.json:

{
  "engines": {
    "node": "20.x"
  }
}

No other environment variables are required — the app has no secrets, no external services, and a mocked OTP/payment flow.

5 Deploy and verify

Click Create Web Service and watch the logs.

  1. Click Create Web Service. Render clones the repo, runs the build command, then the start command.
  2. Watch the Logs tab. A successful boot prints: 
    [db] seeded demo data
[api] AquaLife backend running on http://localhost:10000
  3. When the status turns Live, open the URL Render assigned (e.g. https://aqualife.onrender.com).

⚠ Free services sleep after inactivity A free Web Service spins down after ~15 minutes with no traffic. The next visit triggers a cold start that can take 30–60 seconds. This also restarts the process — and, because of the ephemeral disk, re-seeds the database back to demo defaults.

6 Optional — make data persistent

Only if you need members/check-ins to survive restarts.

The free plan does not include persistent disks. If you want runtime data to survive restarts, you have two options:

  • Add a Render Persistent Disk — requires upgrading to a paid instance type. Mount it at e.g. /data, then point the database there by editing server/db.js to use process.env.DB_PATH || path.join(__dirname, 'data.sqlite') and setting DB_PATH=/data/data.sqlite as an environment variable.
  • Use a hosted database — Render's free PostgreSQL, or an external SQLite-compatible service. This requires changing the data layer from better-sqlite3 to a Postgres client.
For a proof-of-concept, you can skip this entirely. The auto-seed behaviour means every cold start gives you a clean, fully-populated demo — which is often exactly what you want when showing the app.

? Troubleshooting

SymptomFix
Build fails on better-sqlite3 Set NODE_VERSION to 20 (Step 4). Native modules need a supported Node version with build tools — Render's Node runtime includes them.
Page loads but shows a blank screen Confirm the build command includes npm run build. Without dist/, the server has no frontend to serve.
"Cannot GET /" or 404 on routes Same cause — the dist/ folder is missing. Check the build logs for the vite build step completing.
App deploys but never goes Live Ensure the start command is npm start and the app binds to process.env.PORT (it already does).
Data disappeared after a while Expected on the free plan — the disk is ephemeral and the service sleeps. See Step 6.
